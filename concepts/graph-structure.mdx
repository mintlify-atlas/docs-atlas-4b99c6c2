## Graph Representation

The Romania road map is represented as a **weighted, undirected graph** using a Python dictionary structure. This representation provides an intuitive and efficient way to model the network of cities and connecting roads.

## Data Structure Format

The graph is defined in `astar_shortest_path_romania.py:7-28` as:

```python
romania_map = {
    'City': [('Neighbor1', distance1), ('Neighbor2', distance2), ...]
}
```

### Structure Components

- **Keys**: City names (strings representing nodes)
- **Values**: Lists of tuples, where each tuple contains:
  - A neighbor city (string)
  - The road distance to that neighbor (integer, in kilometers)

<Info>
This adjacency list representation is efficient for sparse graphs where each node has relatively few connections, which is typical for road networks.
</Info>

## Edges (Roads)

Edges represent bidirectional roads between cities with associated distances in kilometers.

### Example: Arad's Connections

```python
'Arad': [('Zerind', 75), ('Sibiu', 140), ('Timisoara', 118)]
```

This indicates that from Arad, you can travel to:
- **Zerind**: 75 km
- **Sibiu**: 140 km
- **Timisoara**: 118 km

<Note>
Since the graph is undirected, each connection appears twice (once for each city). For example, if Arad lists Zerind at 75 km, Zerind also lists Arad at 75 km.
</Note>

### Graph Properties

- **Type**: Weighted, undirected graph
- **Nodes**: 20 cities
- **Edges**: Variable per node (ranging from 1 to 4 connections)
- **Weights**: Positive integers representing kilometers
- **Connectivity**: All nodes are connected (no isolated subgraphs)

## Graph Visualization

The implementation includes visualization using NetworkX (astar_shortest_path_romania.py:89-114):

```python
def draw_graph(graph, path=None):
    G = nx.Graph()
    
    # Add nodes and edges from the dictionary
    for city in graph:
        for neighbor, distance in graph[city]:
            G.add_edge(city, neighbor, weight=distance)
    
    # Generate layout and draw
    pos = nx.spring_layout(G)
    nx.draw(G, pos, with_labels=True, node_color='lightblue', 
            node_size=3000, font_size=10, font_weight='bold')
    
    # Highlight the optimal path if provided
    if path:
        path_edges = list(zip(path, path[1:]))
        nx.draw_networkx_edges(G, pos, edgelist=path_edges, 
                               edge_color='r', width=2)
```

<Accordion title="Why use dictionary representation instead of an adjacency matrix?">
The dictionary (adjacency list) representation offers several advantages for this use case:

1. **Space Efficiency**: Only stores existing edges
   - Dictionary: O(V + E) space
   - Matrix: O(V²) space (wastes space for non-existent edges)

2. **Iteration Efficiency**: Easy to iterate over neighbors
   - Directly access `graph[city]` to get all connections
   - No need to scan entire row looking for edges

3. **Dynamic Structure**: Easy to add/remove cities and roads
   - Simply update dictionary keys and values
   - Matrix requires resizing for new nodes

4. **Readable Code**: More intuitive for geographic data
   - Natural representation: "Arad connects to Sibiu at 140 km"
   - Matches mental model of road networks

For dense graphs where most nodes connect to most others, matrices can be more efficient. But for sparse graphs like road networks (where each city connects to only a few others), dictionaries are superior.
</Accordion>

## Bidirectional Consistency

The graph maintains bidirectional consistency—each road is represented in both directions:

```python
# Arad → Sibiu: 140 km
'Arad': [('Sibiu', 140), ...]

# Sibiu → Arad: 140 km (same distance)
'Sibiu': [('Arad', 140), ...]
```

<Info>
This bidirectional representation is essential for undirected graphs and ensures the A* algorithm can traverse roads in either direction with the same cost.
</Info>

## Practical Usage

The graph is passed to the A* search function (astar_shortest_path_romania.py:82):

```python
optimal_path, total_cost = a_star_search(romania_map, 'Arad', 'Bucharest', heuristics)
```

The algorithm uses this structure to:
1. Look up neighbors of the current node
2. Retrieve edge weights (distances)
3. Calculate path costs
4. Find the optimal route through the network

- **Path**: Arad → Sibiu → Rimnicu Vilcea → Pitesti → Bucharest
